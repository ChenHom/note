# 背景與閱讀視角

## 老闆的閱讀動機
Claude 和 OpenAI 只要有系統提示詞流出，我一定會仔細閱讀，觀摩順便學習一線大廠是如何使用文字來駕馭 AI 模型，來服務全球數十億使用者。

## 對提示詞結構的觀察
也可以順便觀察專業的提示詞，他的順序會怎麼安排，像是：
- 禁止類通常放在最前面
- SPO 類型的安排常在中段
- 格式參考與技能推薦通常放在尾段

## 最有洞察的一段
我覺得整個提示詞給我最好的洞察是，Anthropic 很小心地在設計如何引導 AI 給使用者提問，以及教會 AI 什麼樣的問題才是好問題。

那一段很精采，之後可以用後設的角度來思考：
- 為什麼這些問題要這樣排序
- 為什麼先確認上下文，再談變體與 tweak
- 為什麼它不只是叫 AI「問問題」，而是教 AI「問對問題」

## 可延伸的分析框架
未來重讀這類系統提示詞時，可以特別從以下角度拆：
1. **權限邊界**：哪些資訊禁止透露
2. **角色塑形**：模型被要求扮演誰
3. **任務路徑**：先理解、再提問、再探索、再產出、再驗證
4. **提問策略**：哪些問題被視為高品質問題
5. **格式控制**：交付物如何被約束
6. **失敗預防**：提示詞怎麼提前堵住常見爛輸出

## 文章補充後的新觀察
看完外部文章後，這裡可以再補三個判斷：

### 1. 這類 system prompt 的地位，更像產品操作系統而不是單純 prompt
Claude Design 不是只靠一段長文字「叫模型做設計」，而是把：
- 設計上下文蒐集
- 提問策略
- 交付格式
- 驗證流程
- 品牌一致性

串成一套可以落地的工作流協議。

### 2. 真正最值錢的不是表面功能，而是 workflow spec
從媒體文章可看出，Huashu Design 之所以能快速出現，不是因為它單純複製畫面，而是因為 Claude Design 背後那套：
- 先問問題
- 先拿上下文
- 再產出變體
- 再微調
- 再驗證交付

的流程邏輯，本身就已經足夠強，能被 skill 化。

### 3. 「如何問好問題」那段其實是需求蒐集協議
原本看像 prompt engineering 的段落，現在更像是產品需求採樣流程：
- 釐清輸出形式
- 釐清變體維度
- 釐清設計上下文
- 釐清使用者真正重視的價值面向

也就是說，那段精華不只是教模型提問，而是在教模型把模糊需求轉成可執行 spec。
