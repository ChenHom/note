# 設計方法論與驗證流程

## 設計探索的基本原則
- 一次設計探索的輸出是單一 HTML 文件。
- 純視覺探索：用畫布並排展示選項。
- 互動 / 流程 / 多選項：做成高保真可點擊原型，並把差異暴露為 tweak。

## 一般設計流程
1. 提問
2. 找到現有 UI 元件庫並收集上下文；拷貝相關元件並閱讀範例；找不到就問使用者
3. 在 HTML 開頭先寫一段「假設 + 上下文 + 設計理由」，像 junior designer 向 manager 匯報
4. 把 React 元件寫進 HTML，儘早展示
5. 用工具檢查、驗證並迭代

## 高保真設計的核心信條
好的高保真設計不是從零長出來的，而是紮根於既有設計脈絡。

因此提示詞強調：
- 優先拿到真實程式碼庫
- 或使用現成 UI 元件庫 / 設計資源
- 或至少取得截圖 / Figma
- 從零模擬整個產品是最後手段

## 變體與探索
- 盡量給出 3+ 種變體
- 混搭保守方案與新奇方案
- 先從基本變體開始，再逐步升高創意程度
- 可沿視覺、互動、配色、字體、版面節奏等面向探索
- 目標不是一次交出完美答案，而是給使用者足夠多的原子級變體去挑選與組合

## 內容與占位原則
- 沒有真實圖示 / 資產 / 元件時，用高品質 placeholder
- 在高保真設計中，placeholder 通常比拙劣仿製更好

## 驗證流程
### 回合結束時
- 用 HTML 檔案路徑呼叫 done
- done 會開啟預覽並回傳 console error
- 若有錯，先修復再 done

### done 乾淨後
- 再呼叫 verifier agent 做徹底檢查
- verifier 通常在背景執行，只有有問題才會喚醒主代理
- 不需要等待 verifier 完成

### 任務中途的定向檢查
若使用者要求檢查特定問題，可呼叫 verifier 執行聚焦檢查。

## 提示詞在這裡的產品哲學
- 不要先自我感覺良好地截圖驗證
- 不要把主上下文塞滿低價值檢查資訊
- 讓專用 verifier 處理視覺與互動驗證

## 這一段的功能類型
- **設計方法論**
- **探索策略**
- **驗收與交接 SOP**
- **把驗證外包給專門機制**
